# Blueprint MCP Server

Blueprint is an MCP server that gives coding agents a durable project memory layer.

It scans a codebase, groups files by responsibility, and writes compact `.blueprint/` memory that agents can read before inspecting source code. The goal is simple: help LLM coding agents start with the project architecture instead of rediscovering it on every task.

## Install

```bash
npm install -g blueprint-mcp-server
```

Requires Node.js 20 or newer.

## MCP Configuration

Add the server to your MCP client:

```json
{
  "mcpServers": {
    "blueprint": {
      "command": "blueprint-mcp-server"
    }
  }
}
```

For a local checkout:

```json
{
  "mcpServers": {
    "blueprint": {
      "command": "node",
      "args": ["/absolute/path/to/blueprint/dist/index.js"]
    }
  }
}
```

Restart your MCP client after changing the configuration.

## Generated Files

Blueprint writes project memory under `.blueprint/` in the project being analyzed:

| File | Purpose |
| --- | --- |
| `brief.md` | Compact routing index for agents. |
| `groups/*.md` | Architecture notes for each project area. |
| `blueprint-output.json` | Structured project graph generated by the tools. |
| `refresh-scan.json` | Filesystem hash snapshot used for deterministic refreshes. |

Blueprint memory is an orientation layer. Source code remains the source of truth. `.blueprint/` is local generated memory by default; teams can decide whether to commit it or add it to `.gitignore`.

## Initial Workflow

Run these MCP tools from your agent:

1. `blueprint.scan`
2. `blueprint.group` with `mode: "prepare"`
3. Create a compact grouping plan from the prepare output
4. `blueprint.group` with `mode: "apply"`
5. `blueprint.compose`

`blueprint.compose` writes the initial `.blueprint/` memory and patches `AGENTS.md` with a marker-delimited Blueprint snippet without overwriting existing project instructions. If compose returns required next steps for group doc hydration, complete them before treating the setup as finished.

## Maintenance Workflow

When accepted project changes add, move, delete, or substantially change files:

1. Run `blueprint.refresh`.
2. If new files are unassigned, run `blueprint.group.update`.
3. Update only the affected `.blueprint/groups/*.md` notes when architecture, contracts, pitfalls, or test guidance changed.

Do not edit `blueprint-output.json` manually.

## Tools

| Tool | Purpose |
| --- | --- |
| `blueprint.scan` | Builds file inventory and code analysis artifacts. |
| `blueprint.group` | Prepares or applies semantic file grouping. |
| `blueprint.compose` | Writes the final Blueprint output and Markdown notes. |
| `blueprint.refresh` | Refreshes Blueprint state from the current filesystem snapshot. |
| `blueprint.group.update` | Assigns unassigned files or manages empty groups after refresh. |

## Viewer UI

Blueprint ships a lightweight React/Vite viewer, but it does not require a local HTTP server. To view a generated Blueprint in the current project:

```bash
blueprint open
```

`blueprint open` reads `.blueprint/blueprint-output.json` and `.blueprint/groups/*.md`, writes a self-contained `.blueprint/index.html`, then opens it in the default browser.

Use `blueprint open --watch` to keep that static HTML file regenerated while `.blueprint/blueprint-output.json`, `.blueprint/brief.md`, or `.blueprint/groups/*.md` changes. The browser may still need a manual refresh; the watch mode does not start an HTTP server or inject live reload code.

The viewer does not include chat, LLM execution, or HTTP APIs. Vibecoding stays in the terminal/MCP client.

## Language Support

Blueprint uses Tree-sitter analysis for:

- TypeScript / JavaScript
- Python
- Go
- Rust
- Java

Other files are still included in the inventory, but deep symbol/import analysis depends on language support.
